## CRITICAL DEPENDENCY ANALYSIS

### High-Risk Files (Core Logic)
1. **`src/lib/drafting-service.ts`** - Central patent drafting logic, heavily patent-specific
2. **`src/lib/stage-navigation-config.ts`** - Defines entire patent workflow
3. **`src/lib/novelty-search-service.ts`** - Patent search functionality
4. **`src/types/country-profile.ts`** - Type system for jurisdictions

### Database Schema Dependencies
- `Patent` table → rename to `Paper`
- `CountryProfile` → rename to `PublicationVenue`
- `IdeaRecord` → rename to `ResearchTopic`
- Add new tables: `PaperTypeDefinition`, `CitationStyleDefinition`, `Citation`

### API Route Dependencies
- `/api/patents/` → `/api/papers/`
- `/api/country-profiles/` → `/api/publication-venues/`
- Add new routes: `/api/citation-styles/`, `/api/paper-types/`

## MIGRATION RISK ASSESSMENT

### Low Risk
- UI components (can be replaced incrementally)
- Admin interfaces (can be updated after core services)
- Generic utilities (minimal changes needed)

### Medium Risk
- Database schema (requires careful migration)
- API routes (need to maintain backward compatibility during transition)
- Authentication (minimal patent coupling)

### High Risk
- Core services (`drafting-service.ts`, `novelty-search-service.ts`)
- Stage navigation configuration
- Type definitions

---

## RECOMMENDED APPROACH

1. **Start with database schema changes** (Phase 1) - foundation must be solid
2. **Implement core services in parallel** with feature flags for gradual rollout
3. **Replace UI components incrementally** - users can use old patent flows while new paper flows are developed
4. **Maintain dual functionality** during transition with feature flags
5. **Test extensively** - each component replacement could break existing patent workflows

---

## FEATURE FLAG STRATEGY

Add these feature flags to control rollout:
- `ENABLE_PAPER_WRITING`: Enable new paper writing flows
- `ENABLE_CITATIONS`: Enable citation management
- `ENABLE_LITERATURE_SEARCH`: Enable academic search
- `DISABLE_PATENT_DRAFTING`: Gradually disable old patent flows

This allows safe, incremental deployment without breaking existing patent users.
